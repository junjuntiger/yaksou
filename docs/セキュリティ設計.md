# セキュリティ設計 — 薬草みっけ

最終更新: 2026-06-25

---

## 1. 脅威モデル（何から守るか）

| 脅威 | 具体例 |
|------|--------|
| APIキーの漏洩 | GeminiのAPIキーがブラウザに含まれ、悪用される |
| 未認証アクセス | ログインせずにFirestoreのデータを読み書きされる |
| 他人のデータ改ざん | 他ユーザーの投稿を書き換え・削除される |
| APIの乱用 | 大量リクエストで費用が膨らむ |
| 機密情報のGit流出 | APIキー・設定情報がGitHubに上がってしまう |

---

## 2. 現在の対策（実装済み）

### 2-1. APIキー管理

| 項目 | 対策 |
|------|------|
| GEMINI_API_KEY | Vercelの環境変数に設定。コードには含まれない |
| Firebase設定 | `.env` ファイルで管理。`.gitignore` に追加済み |
| Vercel管理画面 | 環境変数はVercelの設定画面からのみ変更可能 |

**フロー**
```
ブラウザ → POST /api/identify → Vercel Function（サーバー）→ Gemini API
                                       ↑
                              ここでAPIキーを使用
                              ブラウザには届かない
```

### 2-2. Firebase Authentication

- メールアドレス＋パスワード認証
- 未ログインユーザーはアプリを一切操作できない
- Firebase SDKがトークンを自動管理（有効期限・リフレッシュ）

### 2-3. Firestoreセキュリティルール（現在）

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /{document=**} {
      allow read, write: if request.auth != null;
    }
  }
}
```

**効果**: 未ログインユーザーは読み書き不可

### 2-4. .gitignore設定

```
.env
.env.local
```

`.env` ファイルはGitHubにpushされないため、APIキー・Firebase設定が流出しない。

## 3. 今後の課題（未対応）

### 3-1. Firestoreルールの強化（優先度：高）

現状は「ログインしていれば他人のデータも書き換えられる」状態。
以下のルールに変更することで自分のデータのみ書き換え可能になる。

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // plants: 読みは誰でも可（ログイン必須）、書きは自分の投稿のみ
    match /plants/{plantId} {
      allow read: if request.auth != null;
      allow create: if request.auth != null
                    && request.resource.data.userId == request.auth.uid;
      allow update, delete: if request.auth != null
                            && resource.data.userId == request.auth.uid;
    }

    // users: 自分のドキュメントのみ読み書き可
    match /users/{userId} {
      allow read, write: if request.auth != null
                         && request.auth.uid == userId;
    }
  }
}
```

### 3-2. API認証の追加（優先度：中）

`/api/identify` は現在誰でも呼び出せる。
FirebaseのIDトークンを検証する処理を追加する。

```typescript
// api/identify.ts に追加予定
import { getAuth } from 'firebase-admin/auth'

const token = req.headers.authorization?.replace('Bearer ', '')
const decoded = await getAuth().verifyIdToken(token)
// decoded.uid で認証済みユーザーを確認
```

### 3-3. Rate Limiting（優先度：中）

Gemini API の無料枠を守るため、1ユーザーあたりの呼び出し回数制限を追加する。
Vercel KV（Redis）または Firestore でカウントを管理する方法がある。

### 3-4. 入力値バリデーション（優先度：低）

Firestoreに保存する前にフィールドの型・長さ・禁止文字をチェックする。

## 4. 環境変数一覧

| 変数名 | 使用場所 | 説明 |
|--------|---------|------|
| `VITE_FIREBASE_API_KEY` | フロントエンド（.env） | Firebase Web APIキー |
| `VITE_FIREBASE_AUTH_DOMAIN` | フロントエンド（.env） | Firebaseの認証ドメイン |
| `VITE_FIREBASE_PROJECT_ID` | フロントエンド（.env） | FirebaseプロジェクトID |
| `VITE_FIREBASE_STORAGE_BUCKET` | フロントエンド（.env） | Firebase Storageバケット |
| `VITE_FIREBASE_MESSAGING_SENDER_ID` | フロントエンド（.env） | Firebase FCM送信者ID |
| `VITE_FIREBASE_APP_ID` | フロントエンド（.env） | Firebase アプリID |
| `GEMINI_API_KEY` | Vercel環境変数（サーバー側のみ） | Google Gemini APIキー |

> **注意**: `VITE_` プレフィックスの変数はビルド時にJSバンドルに含まれる。
> Firebase設定は公開しても比較的安全だが、Firestoreルールでの保護が必須。
> `GEMINI_API_KEY` は絶対に `VITE_` を付けず、サーバー側のみに置くこと。
